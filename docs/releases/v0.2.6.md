# v0.2.6

## Highlights
- Adds `lore update`, a binary-only self-update command for released Lore CLI binaries.
- Adds `lore update --dry-run` to show the current/latest/target plan without replacing the executable.
- Adds `lore update --yes` to skip the interactive confirmation prompt after the same safety checks pass.
- Adds root TUI update awareness: the TUI checks for update availability in the background, surfaces an update banner when a newer release exists, and asks for explicit confirmation before applying the updater.

## What `lore update` does
- Queries GitHub Releases for the latest published Lore CLI release.
- Selects the matching archive for the running platform and architecture.
- Requires the matching `SHA256SUMS` asset and verifies the downloaded archive checksum before any replacement attempt.
- Updates only the active Lore CLI binary.
- Does not modify the Pi runtime, `~/.pi`, extensions, settings, skills, themes, model configuration, provider configuration, or other runtime-managed files.

## Safety model and refusal behavior
- Fails closed for unsupported or unsafe targets.
- Refuses automatic update for dev builds and other unsupported version states.
- Refuses targets when the running executable path does not match the resolved `PATH` entry.
- Refuses symlinked or otherwise unsafe executable targets.
- Refuses updates when the expected platform archive or `SHA256SUMS` asset is missing from the GitHub release.
- Refuses updates on checksum mismatch.
- On Windows, automatic self-update is intentionally unsupported in this release; Lore prints explicit manual replacement guidance instead of attempting in-place mutation.

## Unix update flow
- Downloads the selected release archive and checksum file.
- Verifies archive integrity against `SHA256SUMS` before extraction.
- Extracts the candidate Lore binary from the archive.
- Probes the candidate binary version before replacement when possible.
- Creates a backup of the current binary before replacement.
- Replaces the target binary atomically through a temporary file and rename flow.
- Probes the installed binary again after replacement using the CandidateVersion check.
- Rolls back to the backup automatically if replacement or post-install validation fails.

## TUI update experience
- The root TUI starts a background update check when the shared updater boundary is available.
- While the check is running, the UI remains usable and shows a non-blocking status banner.
- When an update is available, the TUI shows the current and latest versions and lets you select `Update`.
- The TUI asks for explicit confirmation before mutating the binary.
- The TUI keeps the update scope explicit: binary-only, with Pi runtime and `~/.pi` left untouched.

## Upgrade notes
- `lore update` is intended for released binaries on supported Unix platforms.
- `--dry-run` is the safest first step if you want to inspect the planned target and status before mutation.
- `--yes` removes the prompt but does not bypass refusal conditions or checksum verification.
- Windows users should download the matching release archive and replace `lore.exe` manually after exiting Lore CLI.
- If you installed Lore through another distribution channel in the future, keep using that channel unless you explicitly want the standalone binary flow.
