# v0.4.0

## Highlights
- Adds first-class Antigravity/Gemini install support for Lore.
- Configures Gemini/Antigravity to use the remote Lore MCP endpoint directly at `<lore_url>/v1/mcp`.
- Creates a Gemini custom agent profile for Lore at `~/.gemini/config/agents/lore.json`.
- Makes `internal/agentpack` the source of truth for the portable Lore orchestrator system instruction used by generated agent profiles.
- Removes the old local CLI MCP bridge and broker paths.
- Hardens the Windows installer for older Windows PowerShell 5.1 hosts.

## Pi hosted MCP (pi-mcp-adapter)

- `lore install --target pi` now installs hosted Lore MCP via `pi-mcp-adapter` as the default backend.
- The installer writes `~/.pi/agent/settings.json` with a pinned immutable git reference (`git:github.com/nicobailon/pi-mcp-adapter@<sha>`) and `~/.pi/agent/mcp.json` with MCP configuration pointing to the hosted Lore MCP HTTP endpoint (`<lore_url>/v1/mcp`).
- The hosted Lore MCP connects over HTTP/StreamableHTTP to `/v1/mcp` using a bearer token materialized in plaintext in `~/.pi/agent/mcp.json`. The adapter's stdio bridge is not used for the hosted MCP path. This is the same local plaintext-token tradeoff documented for Antigravity MCP config.
- Legacy `extensions/lore-memory.ts` and `extensions/lore-footer.ts` remain available in the repository for rollback or explicit testing via `lore install --target pi --component pi-extensions`, but are not installed by default.
- lore-memory is now dormant by default; the installer only manages `lore-memory` files when the `pi-extensions` component is explicitly selected.

## Antigravity/Gemini
- `lore install --target antigravity` now writes MCP configuration to:

  ```text
  ~/.gemini/config/mcp_config.json
  ```

- The generated MCP config uses the server-backed MCP endpoint:

  ```json
  {
    "mcpServers": {
      "lore": {
        "serverUrl": "<lore_url>/v1/mcp",
        "headers": {
          "Authorization": "Bearer <token>"
        }
      }
    }
  }
  ```

- The bearer token is read from the existing Lore login/keychain flow and written into the Gemini MCP config because Antigravity/Gemini requires headers in that file today. This is intentionally documented as a local plaintext-token tradeoff.
- The installer also writes:

  ```text
  ~/.gemini/config/agents/lore.json
  ```

- The generated Lore agent profile:
  - uses an English description,
  - is marked as the default Gemini agent profile,
  - receives its `systemInstruction` from `internal/agentpack`, plus a small Gemini runtime layer,
  - intentionally omits a fixed `tools` allowlist so `lore-server` remains the authority for tool availability by role and permissions.

## Removed local MCP bridge
- Removes `lore mcp serve`, `lore mcp proxy`, and `lore mcp doctor`.
- Removes the hidden `lore api mcp-call` broker path.
- Removes the local stdio MCP bridge implementation under `internal/mcp`.
- Removes the managed MCP auth-file flow, including `lore_mcp_auth.json` generation.
- Updates Pi memory integration to avoid the old CLI MCP broker path.

## Agentpack source of truth
- Adds an agentpack-owned orchestrator instruction renderer.
- Antigravity/Gemini agent instructions are now generated from agentpack-owned Lore persona/workflow semantics instead of a separate install-only asset.
- This reduces drift between Pi-managed configuration, Antigravity/Gemini configuration, and future managed harnesses.

## Windows installer hardening
- Declares PowerShell 5.1 compatibility in `scripts/install.ps1`.
- Adds a TLS 1.2 bootstrap for older Windows PowerShell hosts.
- Uses a web-request compatibility wrapper with `-UseBasicParsing` where needed.
- Resolves `-Version latest` through the GitHub REST API with a `User-Agent` header.
- Keeps the existing stricter SHA256 verification and PATH behavior.
- Updates Windows installer smoke coverage for the latest-version path.

## Upgrade notes
- Any scripts or integrations that used the removed local MCP bridge commands must switch to the server-backed MCP endpoint at `/v1/mcp`.
- Antigravity/Gemini users should rerun `lore install --target antigravity` after upgrading.
- On older Windows PowerShell hosts, prefer a pinned release version when running the install script.
- `go test ./...` is currently known to fail in `internal/update` due to an unrelated release-asset selection expectation; focused tests for this release passed.

## Validation summary
- `go test ./internal/agentpack ./internal/install ./internal/cli ./internal/httpclient ./internal/tui`
- PowerShell installer changes received static checks, but PowerShell runtime execution was not available on this host.
